# Publication boundary

Brilliant Brainstorm Intelligence separates **continuous deployment** from **continuous disclosure**.

Public surfaces may update after human merge. Nothing becomes public merely because it exists in a private system or on a local machine.

## Safe for automatic publication

```text
Public repository metadata
Published releases
Public issue and discussion links
Public CI results
Public licenses
Approved public project status
Approved article metadata
Public roadmap links
Verified public deployment URLs
```

Source of approved facts: `Brilliant-Brainstorm-Intelligence-LLC/bbi-public-proof-registry`

## Human approval required

```text
Biography changes
Role positioning
Architecture narratives
Case studies
Business-impact claims
Speaking availability
Résumé changes
Certification claims
Screenshots
Company offers
```

## Never publish

Do not publish private operating-system source or internals, private agent-prompt systems, private financial planning material, private research internals, private prompts, private roadmaps, private repositories, client information, employer-confidential information, personal finances, legal or tax records, credentials, secrets, private contact information, local file paths, or unsupported metrics.

Named private systems and personal operating records remain outside the public boundary. Governance documents may name excluded categories to forbid them; they must not include private content.

## Public Proof Gate

Every public update should pass:

```text
Build / validation
Privacy boundary scan
Claim and evidence validation
Public-source validation
External-link validation where applicable
Human merge to protected main
```

Fail closed when:

- A source URL is private or inaccessible
- A claim has no evidence
- A forbidden path or credential-shaped secret appears
- A record is not explicitly marked public
- A project is labeled released without a release
- A planned project is presented as completed

## One-way publication

```text
Private work
→ candidate public summary
→ sanitization
→ evidence review
→ Public Proof Gate
→ public registry
→ profile/site update PR
→ CI
→ human merge
→ automatic deployment
→ live verification
```

Public workflows must not:

```text
Clone private operating repositories
Read private repositories by default
Scan personal cloud folders
Query private planning systems as publication sources
Use a token with broad private-repository access
Publish local documents
Generate case studies directly from internal records
```

There is **no reverse checkout or automated ingestion** from private operating systems into public surfaces.

## Auto-assign and ceremonial activity

- Do not auto-assign every new issue during the sole-maintainer phase
- Do not create fake pull requests to clear onboarding cards
- Do not invite placeholder members
- Do not install starter CI over working custom workflows

Issue flow:

```text
New issue → Roadmap Inbox → Triage → Assign when Ready or In Progress
```
